# Review: wezterm-server Workspace Awareness Phase 1 Implementation

## Summary Assessment

This review covers the Phase 1 implementation of the wezterm-server workspace awareness proposal: the actual code changes to `install.sh`, `devcontainer-feature.json`, `README.md`, and the new `workspace_config` test scenario.
The implementation is correct in all material aspects.
Shell heredoc quoting is handled properly, the feature metadata is valid, and the README accurately documents the new behavior.
The most notable gap is a missing test for the baked `_REMOTE_USER` value, which the proposal explicitly specified.

Verdict: **Accept** with non-blocking suggestions.

## Section-by-Section Findings

### install.sh lines 85-102: wezterm.lua heredoc

The `wezterm.lua` heredoc uses a **quoted** delimiter (`<< 'WEZTERM_CONFIG'`), preventing all variable expansion.
This is correct: the Lua source is written verbatim.

Generated output:

```lua
local wezterm = require("wezterm")
local config = wezterm.config_builder()

local workspace = os.getenv("CONTAINER_WORKSPACE_FOLDER")
if workspace then
  config.default_cwd = workspace
end

return config
```

The `if workspace then` guard provides clean graceful degradation when the env var is absent.

**Finding: correct.** No issues.

### install.sh lines 104-118: entrypoint.sh heredoc

The entrypoint heredoc uses an **unquoted** delimiter (`<< ENTRYPOINT`), which means all `$` references are expanded at install time.
This is intentional for baking `$_REMOTE_USER` and `$WEZTERM_SERVER_DIR`.

Line-by-line expansion trace (assuming `_REMOTE_USER=node`):

| Source (install.sh) | Generated (entrypoint.sh) | Correct? |
|---|---|---|
| `\$(id -u)` | `$(id -u)` (runtime eval) | Yes |
| `"${_REMOTE_USER}"` on line 112 | `"node"` (baked) | Yes |
| `$WEZTERM_SERVER_DIR` on lines 113, 115 | `/usr/local/share/wezterm-server` (baked) | Yes |
| `${_REMOTE_USER}` at end of line 113 | `node` (baked, `su` target) | Yes |

The single quotes around the `su -c` argument on line 113 are literal characters in the heredoc output.
At runtime they protect the baked command string from the entrypoint shell, which is correct since `$WEZTERM_SERVER_DIR` is already a literal path.

**Finding: correct.** No issues.

### install.sh line 107: `_REMOTE_USER` scoping

Line 79 assigns `_REMOTE_USER="${_REMOTE_USER:-root}"` inside the `if [ "$CREATERUNTIMEDIR" = "true" ]` block.
Line 107 assigns `_REMOTE_USER="${_REMOTE_USER:-root}"` unconditionally.

When `CREATERUNTIMEDIR=false`, line 79 is skipped.
Line 107 ensures `_REMOTE_USER` is always set before the entrypoint heredoc.
When `CREATERUNTIMEDIR=true`, line 107 is a harmless no-op.

**Finding: correct.** The redundancy is intentional and defensive.

### install.sh: `su -c` environment inheritance

The entrypoint uses `su -c '...' <user>` (without `-l` or `-`) to drop privileges.
On Debian-based systems (the only supported distro family), `su` without a login flag preserves the parent environment.
This means `CONTAINER_WORKSPACE_FOLDER` from `containerEnv` (injected as `docker run -e`) will be available to `wezterm-mux-server`.

The `wezterm-mux-server` reads the config file at startup, calls `os.getenv("CONTAINER_WORKSPACE_FOLDER")` in Lua, and sets `default_cwd`.
The `--daemonize` flag causes the server to fork, but the forked process inherits the environment from its parent.
This chain is correct.

**Finding: non-blocking concern.** The behavior is correct for Debian.
If RPM-based distro support is added in the future, `su` behavior may differ due to PAM configuration.
A comment in the heredoc body noting that `-l` is intentionally omitted to preserve env would prevent a well-meaning future edit.

### Test coverage: workspace_config.sh

The test verifies five things:

1. `wezterm.lua` exists at the expected path.
2. `entrypoint.sh` exists at the expected path.
3. `entrypoint.sh` is executable.
4. `wezterm.lua` contains the `os.getenv("CONTAINER_WORKSPACE_FOLDER")` call.
5. `entrypoint.sh` contains `--config-file /usr/local/share/wezterm-server/wezterm.lua`.

**Missing from proposal spec:** Section 1.4 explicitly requires "Verify entrypoint.sh contains the baked `$_REMOTE_USER` value."
This test is not present.
On the devcontainer base images used by the scenarios (`mcr.microsoft.com/devcontainers/base:debian`), `_REMOTE_USER` defaults to `vscode`.
A check like `grep -q 'vscode\|node' /usr/local/share/wezterm-server/entrypoint.sh` would verify the baking happened correctly and caught any regression where the heredoc quoting changed.

Without this test, a change to the heredoc (e.g., quoting the delimiter) would silently bake literal `${_REMOTE_USER}` instead of the resolved username, and the test suite would not catch it.

**Finding: non-blocking.** The risk is low (the heredoc quoting is correct and unlikely to be accidentally changed), but the proposal explicitly called for this test.

### Test coverage: scenarios.json

The `workspace_config` scenario is correctly defined with a standard base image and default feature options.
It properly integrates with the devcontainer features test framework.

The E2E behavioral tests (mux server process running, pane cwd verification, `su -c` env inheritance) were marked "may require E2E" in the proposal and are appropriately deferred from this Phase 1 scope.

**Finding: correct.** No issues.

## Verdict

**Accept.**

The Phase 1 implementation is correct.
Heredoc quoting handles the install-time-baking vs runtime-evaluation distinction properly.
The feature metadata is valid and follows established patterns.
The README is accurate and complete for the new scope.
Test coverage verifies the essential artifacts, with one non-blocking gap.

## Action Items

1. [non-blocking] Add a test assertion in `workspace_config.sh` to verify the baked `_REMOTE_USER` appears in `entrypoint.sh`. The devcontainer base image sets `_REMOTE_USER=vscode`, so: `check "entrypoint.sh bakes remote user" grep -q 'vscode' /usr/local/share/wezterm-server/entrypoint.sh`. This fulfills the proposal's explicit test requirement from section 1.4.
2. [non-blocking] Add a comment in the entrypoint heredoc body (around line 112-113) noting that `su` is intentionally used without `-l` to preserve the parent environment, specifically `CONTAINER_WORKSPACE_FOLDER`. This prevents a future maintainer from adding `-l` and silently breaking workspace awareness.
3. [non-blocking] When RPM-based distro support is added, verify `su -c` env preservation behavior, or consider switching to `runuser -u <user> -- <command>` for more predictable cross-distro semantics.
