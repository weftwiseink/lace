# Devcontainer Workspace Path Migration

> BLUF: Migrate the container workspace root from `/workspace/lace` to `/workspaces/lace` and eliminate the Dockerfile's `COPY . .` to fix stale-file shadowing that causes Claude Code "external imports" warnings on startup.
> The migration touches the Dockerfile, devcontainer.json, two default values in `workspace-layout.ts`, and dozens of test assertions.
> No runtime behavior changes: the bind mount provides all source at runtime, so removing the COPY is safe.

## Objective

Eliminate the stale-file shadowing problem documented in `cdocs/reports/2026-03-20-devcontainer-workspace-path-audit.md`.
Align with the standard devcontainer convention of `/workspaces/<project>`.

## Background

The Dockerfile uses `WORKDIR /workspace` and `COPY . .`, baking a full repo snapshot into the image at `/workspace/`.
At runtime, the live repo is bind-mounted at `/workspace/lace/main`.
Claude Code walks up from `/workspace/lace/main` and finds stale `CLAUDE.md` and `.claude/rules/` at `/workspace/`, triggering "external imports" warnings.

The standard devcontainer convention uses `/workspaces/<project>`.
This project uses `/workspace/lace` instead.

See the audit report for inode-level evidence of the duplication.

## Proposed Solution

### 1. Isolate Dockerfile build path

Move the dependency-install COPY to an isolated build directory (e.g., `/build`).
Remove `COPY . .` entirely: the bind mount provides all source at runtime.

```dockerfile
# Build-time dependency cache (isolated from runtime workspace)
WORKDIR /build
COPY --chown=${USERNAME}:${USERNAME} package.json pnpm-lock.yaml pnpm-workspace.yaml ./
RUN pnpm install --frozen-lockfile

# Runtime workspace (created empty, populated by bind mount)
RUN mkdir -p /workspaces/lace /home/${USERNAME}/.claude ${COMMAND_HISTORY_PATH} && \
    chown -R ${USERNAME}:${USERNAME} /workspaces/lace /home/${USERNAME}/.claude ${COMMAND_HISTORY_PATH}

WORKDIR /workspaces/lace
```

> NOTE(opus/workspace-path-migration): The `pnpm install` at build time caches `node_modules` in `/build`.
> At runtime, the bind mount provides the full repo and `pnpm install` runs again via `postCreateCommand` or manually.
> The build-time install exists only to pre-warm the pnpm store in the image layer.
> If node_modules must be at the workspace path, a symlink or store-dir config can bridge them.

### 2. Migrate mount target to `/workspaces/lace`

Update `customizations.lace.workspace.mountTarget` from `/workspace/lace` to `/workspaces/lace`.
Update the two default fallback values in `workspace-layout.ts`.

### 3. Update devcontainer.json fallback comments

The commented-out `workspaceMount` and `workspaceFolder` lines reference `/workspace`.
Update to `/workspaces/lace`.

## Important Design Decisions

**Why `/workspaces/lace` (plural `workspaces`)?**
This is the standard devcontainer convention (`/workspaces/<project>`) used by VS Code, GitHub Codespaces, and the devcontainer CLI.
Aligning with it reduces friction for contributors using standard tooling.

**Why remove `COPY . .` instead of just changing the path?**
The COPY serves no runtime purpose: the bind mount provides all source files.
It existed to make `pnpm install` work during the image build (which needs `package.json` etc.).
Scoping the COPY to only dependency files eliminates the stale-file problem entirely.

**Why an isolated `/build` directory?**
Putting the dependency COPY at `/workspaces/lace` would recreate the same shadowing problem.
An isolated path ensures build artifacts never overlap with the runtime workspace.

## Edge Cases / Challenging Scenarios

**Existing containers.**
Users with existing containers will need to rebuild after this change.
The `mountTarget` change means the old bind mount path won't match.
This is a breaking change for existing containers, but `lace up` handles rebuilds.

**pnpm store location.**
The build-time `pnpm install` populates the store at `/build/node_modules`.
At runtime, `pnpm install` in the workspace will use the pnpm content-addressable store (shared), so most packages are already cached in the image layer.
No performance regression expected.

**Hardcoded `/workspace` in user settings.**
Users who have customized their `~/.config/lace/settings.json` with paths referencing `/workspace` will need to update.
This is unlikely since `mountTarget` is typically only set in `devcontainer.json`.

**`.lace/devcontainer.json` (generated config).**
This file is generated by `lace up` and contains hardcoded `/workspace/lace` paths.
It regenerates on `lace up`, so no manual update is needed.
